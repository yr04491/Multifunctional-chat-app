## 1. プロジェクト全体像

### このアプリは何か
Next.js + Firebase で作られた多機能チャットアプリです。
時空カオス（メッセージの遅延・並び変え）や暗号化などのギミックを備えています。

- **フロントエンド:** Next.js 16 / React 19 / TypeScript / Tailwind CSS
- **バックエンド/DB:** Firebase（Firestore）
- **主要ファイル構成:**
  ```
  src/
  ├── app/               # Next.js ページ
  ├── components/
  │   ├── ChatInterface.tsx   # チャット画面のメインコンポーネント
  │   └── OptionsModal.tsx    # 設定モーダル（敬語ONスイッチがここにある）
  └── lib/
      ├── honorificRules.ts   # 【今回の主対象】敬語変換エンジン
      ├── firebase.ts         # Firebase 初期化
      ├── useChat.ts          # チャット送受信ロジック
      └── useAuth.ts          # 認証ロジック
  ```

### 敬語変換機能の位置づけ
ユーザーが「敬語モード」をONにした状態でメッセージを送ると、
`honorificRules.ts` 内の `convertToTeineigo(text)` が呼ばれ、
送信前にカジュアルな日本語が丁寧語（です・ます調）に自動変換される。

**呼び出し元:** `src/components/ChatInterface.tsx`
**外部インターフェース（変えてはいけない）:** `export function convertToTeineigo(text: string): string`

## 3. 現状の問題点

### 現在のアーキテクチャ（課題あり）
`honorificRules.ts` は現在、**「正規表現による全文一括置換」** を繰り返す方式で実装されている。

```
入力テキスト（String）
    → phraseRules の replace を全件実行
    → pronounRules の replace を全件実行
    → conjunctionRules の replace を全件実行
    ...（8種類のルールセットを順番に実行）
    → __DESU__ などのプレースホルダーを最後に一括で本物の日本語に戻す
    → 出力テキスト（String）
```

**この方式の問題点:**
- ルール A の変換結果に、後のルール B が誤って反応する（二重変換）を防ぐため、`__DESU__` `__MASU__` などの「プレースホルダー文字列（黒魔術）」が必要になっている
- 「走る」→「走ります」が正しいのに「走ます」になるバグが発生している（後述）
- ルールの順番を変えると他の変換が壊れる可能性があり、保守が難しい

### 既知バグ（修正必須）
| 入力 | 現在の誤った出力 | 正しい出力 | 原因 |
|---|---|---|---|
| 早く走るが | 早く**走ます**が | 早く**走ります**が | 五段活用例外の未対応 |
| 帰る | **帰ます** | **帰ります** | 同上 |
| 知る | **知ます** | **知ります** | 同上 |
| 切る | **切ます** | **切ります** | 同上 |
| 喋る | **喋ます** | **喋ります** | 同上 |

**根本原因:** 一段動詞判定ルール `([いきし...れ])る → ます` が、
見た目が似ている五段動詞（走る・帰る等）を誤って一段動詞と判定している。
新アーキテクチャでは辞書で明示的に管理することで根本解決する。

## 4. 新しいアーキテクチャ（何をどう変えるか）

プログラミング言語のコンパイラと同じ **3フェーズ・パイプライン** に変える。
外部ライブラリは一切使わず、TypeScript の標準機能だけで実装する。

```
入力テキスト
    ↓
[フェーズ1] Lexer（字句解析）
    文章を左から右へ1文字ずつスキャンし、
    「単語（トークン）のオブジェクト配列」に分割する
    ↓
[フェーズ2] Parser（構文評価・変換）
    トークン配列をループし、
    前後のトークンを見て丁寧語に書き換える
    ↓
[フェーズ3] Generator（文字列生成）
    変換済みトークンを join('') で結合して完成
    ↓
出力テキスト（丁寧語）
```

### トークンのデータ構造（イメージ）
```typescript
interface Token {
  surface: string; // 元の文字列（例: "走る"）
  pos: string;     // 品詞（例: "verb", "noun", "particle", "slang"）
  stem?: string;   // 動詞の語幹（例: "走"）
  conjugationType?: "godan" | "ichidan"; // 活用タイプ（五段 or 一段）
}
```

### Lexer の動作（最長一致法）
辞書（JSON）に登録された単語を、文字列の先頭から最も長くマッチするものを優先して選ぶ。
辞書に載っていない単語は文字種（漢字の連続・ひらがなの連続など）でパッキングして `unknown` 扱い。

```
"俺走る" を処理する場合:
  1. 先頭を見る → 辞書に "俺" がある（pronoun）→ Token { surface: "俺", pos: "pronoun" }
  2. 次を見る    → 辞書に "走る" がある（verb, godan）→ Token { surface: "走る", pos: "verb", stem: "走", conjugationType: "godan" }
  3. 完成 → [ { "俺", pronoun }, { "走る", verb, godan } ]
```

### Parser の動作（文脈を見た変換）
```
トークン配列: [ 俺(pronoun), が(particle), 走る(verb:godan), 。(punct) ]

→ "俺" + pronoun → "私" に変換
→ "走る" + godan + 「次が句読点=文末」 → "走ります" に変換
→ 結合して「私が走ります。」の完成
```

## 9. よくある質問（FAQ）

**Q: 外部ライブラリを使えばもっと楽に作れるのでは？**
A: 意図的に使っていません。このプロジェクトの核心コンセプトは「外部 AI もライブラリも使わず、TypeScript 標準機能だけで自作する」ことです。軽量・高速・コストゼロ・プライバシー安全が実現できます。

**Q: 尊敬語や謙譲語には対応しないの？**
A: 今回はスコープ外です。「主語が誰か」を判定するには文脈の複数文にわたる解析が必要で、ルールベースでの精度担保が現実的でないためです。

**Q: 辞書にない単語が来たらどうなる？**
A: `unknown` トークンとして扱い、そのまま出力します。変換ミスより「変換しない」方がマシという設計方針です。

**Q: バグを見つけたら？**
A: Phase 5 完成後はアプリ内のバグ報告ボタンから報告してください。それまでは `test_honorific.ts` にケースを追加して、`node test_honorific.ts` で再現確認してください。
